# 三维地质数据对接需求文档

> **用途**:向算法/建模团队说明前端展示系统需要什么样的三维地质数据,以便把团队的预测成果接入「多源勘察数据联动展示系统」进行三维可视化。
>
> **适用场景**:内部对接沟通(比赛/课设项目)。
> **对接目标工程**:XX 铁路隧道 K12+000 ~ K13+000 段(与建模区域为同一工程)。

---

## 一、为什么要这份文档

目前展示系统已具备:

- ✅ **地表三维点云**(Three.js 渲染,坡度着色,可飞行定位)
- ✅ **二维地质资料**(钻孔柱状图、物探电阻率剖面、钻孔地层条形图)
- ✅ **风险分析链路**(多源证据 → 雷达图 → Word/MD/HTML 报告)

**缺少的是「连续的三维地下地质结构可视化」** —— 现在地下地质全是离散二维的(6 个钻孔柱 + 物探断面),无法像团队成果那样呈现「整个区域的三维土壤/岩性分类体」。

团队的成果(三维土壤层序预测模型)正好填补这个缺口。本文档约定双方的数据交付接口。

## 三、交付物 1:三维分类体(核心)

### 3.1 格式选择:体素网格(Voxel Grid)

**请交付规则三维网格**,每个体素格包含一个类别编号。

> **通俗解释**:把地下空间想象成一个巨大的「魔方」,切成一个个小立方格子,每个格子里写一个数字代表这块是什么土/岩。前端拿到后可以渲染成三维体(类似你们效果图里那种立体的土壤分类块)。

### 3.2 推荐文件格式(任选其一,优先 JSON)

#### 方案 A:JSON(推荐,最省事)

文件名:`voxel_grid.json`

```json
{
  "format": "voxel_grid",
  "version": "1.0",
  "shape": [200, 200, 50],
  "spacing_m": [10.0, 10.0, 1.0],
  "origin_xyz": [0.0, 0.0, -50.0],
  "axis_order": ["x", "y", "z"],
  "nodata": -1,
  "data": "base64 或 flat 整数数组",
  "data_dtype": "int8 或 int16",
  "categories_ref": "categories.json"
}
```

字段说明:

| 字段 | 含义 | 示例 |
|---|---|---|
| `shape` | 三个方向的格数 `[nx, ny, nz]` | `[200, 200, 50]` 表示 X 方向 200 格、Y 方向 200 格、Z 方向 50 格 |
| `spacing_m` | 每格的物理尺寸(米) `[dx, dy, dz]` | `[10, 10, 1]` 即每格 10m×10m×1m |
| `origin_xyz` | 网格**左下后角**在工程坐标系中的位置 `[x0, y0, z0]` | `[0, 0, -50]` |
| `axis_order` | 数据维度的排列顺序 | `["x","y","z"]` |
| `nodata` | 无数据/未预测的填充值 | `-1` |
| `data` | 体素值数组,长度 = nx×ny×nz | 见下方「数据排列」 |
| `categories_ref` | 类别编码表文件名 | 指向交付物 2 |

**数据排列**:`data` 按行优先(flat),索引公式为:
```
index = (iz * ny + iy) * nx + ix
```
即最外层是 Z,中间是 Y,最内层是 X。**如果你们的习惯不同,请在 `axis_order` 里写清楚,我这边适配。**

#### 方案 B:NumPy `.npy` / `.npz`(大数据推荐)

如果体素量大(>5000 万格),JSON 会很大。请改用:
- `voxel.npy`:三维 numpy 数组,`dtype=int8/int16`,`shape=(nz, ny, nx)`(注意 Z 在前)
- `meta.json`:同方案 A 里的 shape/spacing/origin/nodata/categories

#### 方案 C:其它(需预先沟通)

如果团队只能产出 **GeoTIFF(多波段)**、**VTK**、**UBC 网格** 等,请提前告知,我这边写对应的解析适配器。**不建议交付纯图片(PNG 截图),图片无法做交互式三维渲染。**

### 3.3 数据排列示意图

```
          Z (深度方向, 向下为正)
          ↑ 索引 iz: 0..nz-1
          |
          +---------+
         /         /|
        /  data   / |
       +---------+  |
       |  (体素) |  +
       |         | /
       +---------+
      iy=0..ny-1 → Y
      /
     +--- ix=0..nx-1 → X

origin_xyz 指向 (ix=0, iy=0, iz=0) 这个角的工程坐标
某体素 (ix,iy,iz) 的中心工程坐标 =
    ( origin_x + (ix+0.5)*dx,
      origin_y + (iy+0.5)*dy,
      origin_z + (iz+0.5)*dz )
```

## 五、交付物 3:坐标变换参数(关键)

### 5.1 两个坐标系

| 坐标系 | 用途 | 范围 |
|---|---|---|
| **工程坐标系(本系统)** | 展示系统内部统一坐标 | X: 0~1000m,Y: 0~800m,高程 ~920~1080m |
| **团队建模坐标** | 团队三维模型坐标 | 请团队填写:NAP 基准?Local X/Y?范围? |

### 5.2 现有系统的坐标基准(供团队对齐)

```
工程区域: 1000m(X,东西) × 800m(Y,南北)
对应里程: X=0 → K12+000 , X=1000 → K13+000
DEM 原点(左下角): 工程坐标 (0, 0)
DEM 高程范围: 约 920m ~ 1080m(绝对高程)
DEM 像素分辨率: 2m/像素
高斯坐标参考(假定的标定值): ORIGIN_X=500000, ORIGIN_Y=3400000
```

### 5.3 请团队提供

由于是**同一工程**,团队的三维模型坐标应该能对上工程坐标。请提供以下任一种变换关系:

**情况 A:如果团队坐标就是工程坐标(最简单)**
```
直接说明:体素的 origin_xyz 已经是工程坐标即可。
```

**情况 B:如果团队用了不同的原点/高程基准(如 NAP)**
请给出 7 个参数(或简化为平移+比例):
```
平移:   team_x = eng_x + dx       (dx = ? )
        team_y = eng_y + dy       (dy = ? )
        team_z = eng_elev + dz    (例如 NAP 换算: dz = ? )
旋转:   (一般工程段可忽略)
比例:   (一般 1:1)
```
**最少**也要给出:**高程基准换算值**(NAP 0m 对应工程绝对高程多少米)。

> ⚠️ 坐标对不齐会导致三维地质体和地表点云、钻孔、风险区全部错位,这是对接成败的关键,请务必确认。

## 七、验收标准(团队交付后,我会这样检查)

1. **能加载**:体素数据能被正确解析成 `(nz,ny,nx)` 数组,无明显格式错误。
2. **对得齐**:三维地质体落在地表点云下方、与 6 个钻孔柱位置吻合(误差 < 1 个格距)。
3. **看得懂**:类别编码表与体素数值范围一致,每类有中文名和颜色。
4. **可渲染**:数据量适中(建议总格数 < 2000 万),前端 WebGL 能流畅旋转。
